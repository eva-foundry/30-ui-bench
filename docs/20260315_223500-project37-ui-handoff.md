## FK And Domain UX Contract

Project 30 should not treat foreign keys as raw ids that happen to appear in a
table cell. If a screen is driven by related model objects, the UI should turn
those relationships into navigation and context.

Use these rules:

- prefer `GET /model/domain-views` and `GET /model/layer-metadata/` before
  falling back to raw layer exploration
- render FK fields as navigable links when the target object has its own page,
  drawer, or detail panel
- render parent/child relationships as grouped lists, breadcrumbs, dependency
  chains, or master/detail panels rather than repeating opaque ids
- where a related object is not yet seeded, show the id as a fallback with a
  clear `pending runtime seed` or `not yet available` state
- where the same relation appears in multiple places, promote it into a shared
  UI pattern instead of re-implementing bespoke FK widgets on each page

Initial FK-driven patterns expected from this packet:

- workflow run -> definition, stages, transitions, signals, gate results
- agent memory -> agent identity, procedures, lifecycle events, evidence,
  verification records
- work-factory and service views -> capability, reusable pattern, SLO, breach,
  run, remediation, and lifecycle relationships

This packet now assumes that screens should help agents and operators reason by
ontology category and domain view first, with raw layer rows only as a fallback
inspection mode.
